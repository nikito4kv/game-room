# План разработки «Game Room»

> Пошаговый roadmap по [SPEC.md](SPEC.md). Делаем от простого к сложному:
> каждый этап даёт что-то рабочее, что можно проверить. Подходит для разработки
> с помощью нейросети — этапы маленькие и с чёткой целью.
> Дата: 2026-06-23.

---

## Как пользоваться планом

- Иди по этапам сверху вниз. Не начинай следующий, пока текущий не «проверяется».
- В конце каждого этапа есть блок **«Как проверить»** — это твоя приёмка.
- Каждый этап — отдельный коммит (а лучше отдельная ветка) в Git.
- Промпты нейросети формулируй на один этап за раз, давая ей `SPEC.md` как контекст.

---

## Этап 0. Подготовка (фундамент)

**Цель:** пустой, но запускающийся проект Next.js + готовые доступы к LiveKit.

Задачи:
1. Создать проект Next.js (TypeScript, App Router):
   ```bash
   npx create-next-app@latest . --typescript --app --eslint
   ```
2. Завести бесплатный аккаунт **LiveKit Cloud**, создать проект, получить:
   `LIVEKIT_URL`, `LIVEKIT_API_KEY`, `LIVEKIT_API_SECRET`.
3. Создать файл `.env.local` и положить туда ключи. Добавить `.env.local` в `.gitignore`.
4. Установить SDK:
   ```bash
   npm install livekit-client @livekit/components-react livekit-server-sdk
   ```
5. Залить первый коммит на GitHub (репозиторий уже привязан).

**Как проверить:** `npm run dev` поднимает пустую страницу на `localhost:3000`,
ключи LiveKit лежат в `.env.local` и НЕ попали в git.

## Этап 1. Каркас комнат (создание и вход по коду)

**Цель:** можно создать комнату, получить код и зайти в неё (пока без звука).

Задачи:
1. **Лендинг** (`/`): кнопки «Создать комнату» и «Войти по коду», поле ника.
2. **Серверный эндпоинт генерации токена** (Route Handler, напр. `app/api/token/route.ts`):
   принимает имя комнаты + ник, возвращает токен LiveKit. Ключ/секрет — только на сервере.
3. **Создание комнаты:** генерация короткого кода комнаты; запись метаданных
   (название, флаг публичности, хэш пароля, кто хост) в `metadata` комнаты.
4. **Проверка пароля** на сервере перед выдачей токена.
5. **Страница комнаты** (`/room/[code]`): подключение к LiveKit, показ списка участников.

**Как проверить:** с двух вкладок браузера заходишь по одному коду — оба видят
друг друга в списке участников комнаты.

---

## Этап 2. Голос + управление звуком (ядро)

**Цель:** участники слышат друг друга, могут управлять микрофоном и звуком.

Задачи:
1. Запрос доступа к микрофону, публикация аудио-трека в LiveKit.
2. Кнопка **вкл/выкл своего микрофона** (mute/unmute).
3. Кнопка **вкл/выкл общего звука** (заглушить приём всех).
4. **Индикация говорящего** (подсветка активного участника).
5. Обработка ошибок: нет доступа к микрофону, обрыв связи и переподключение.

**Как проверить:** два устройства (или вкладки с разными микрофонами) слышат друг
друга; кнопки mute работают; видно, кто говорит.

---

## Этап 3. Демонстрация экрана

**Цель:** участник может показать свой экран комнате.

Задачи:
1. Кнопка «Демонстрация экрана» → публикация screen-share трека (getDisplayMedia).
2. Отображение чужого экрана крупно в центре комнаты.
3. Остановка демонстрации; обработка случая «закрыл шаринг из системного окна».

**Как проверить:** один участник шарит экран — остальные видят его картинку,
останавливается корректно.

---

## Этап 4. Доска тактик — базовая (P0)

**Цель:** общая доска, где рисунки видят все в реальном времени.

Задачи:
1. Холст (canvas) с **свободным рисованием:** кисть, выбор цвета, ластик, очистка.
2. **Фон-карта:** загрузка картинки и рисование поверх неё.
3. **Синхронизация в реальном времени** через data-канал LiveKit (рисунки летят всем).
4. Кто может рисовать — управляется хостом (см. этап 5).

**Как проверить:** один рисует — у других сразу появляются те же линии; смена фона
видна всем.

---

## Этап 5. Модерация хоста + настройки устройств

**Цель:** хост управляет комнатой; каждый настраивает свои устройства.

Задачи:
1. **Права хоста:** кикнуть/«забанить» (в пределах сессии), заглушить чужой
   микрофон, передать права хоста, управлять доской.
2. **Авто-передача прав хоста**, если хост вышел.
3. **Авто-удаление комнаты**, когда вышел последний участник.
4. **Настройки устройств:** выбор микрофона, выбор устройства вывода, громкость
   (сохранение выбора в localStorage).

**Как проверить:** хост кикает участника — тот вылетает; заглушает чужой микрофон;
смена микрофона в настройках работает; пустая комната исчезает.

## Этап 6. Стиль CS 1.6 (визуал)

**Цель:** интерфейс выглядит как CS 1.6, но остаётся удобным.

Задачи:
1. Тема: зелёные «лифтовые» панели/рамки, угловатые окна, пиксельный шрифт в
   заголовках/акцентах; читаемый шрифт для мелкого текста.
2. Применить стиль к лендингу, комнате, панели управления, доске, настройкам.
3. Аккуратные состояния: загрузка, ошибки, запрос доступа к микрофону/экрану.

**Как проверить:** все экраны в едином ретро-стиле, при этом всем удобно
пользоваться (кнопки понятны, текст читается).

---

## Этап 7. P1 — улучшения после MVP

**Цель:** то, что делает продукт полнее (не блокирует запуск).

Задачи:
1. **Доска — расширенная:** фигуры и стрелки, текст, перемещаемые метки игроков.
2. **Публичный список комнат:** витрина комнат с флагом «публичная» (через
   LiveKit `listRooms` + фильтр по metadata), поиск/фильтр.
3. **«Память» браузера:** запоминание ника и настроек.
4. Индикатор качества связи.

**Как проверить:** на лендинге виден список публичных комнат; на доске можно
ставить стрелки/метки; ник подставляется автоматически при повторном входе.

---

## Что НЕ делаем сейчас (из спеки)

Аккаунты, мобильная версия, текстовый чат, запись/сохранение доски, монетизация,
полноценный бан. Это P2 — после того, как MVP заработает и появятся пользователи.

---

## Порядок приоритетов кратко

1. **Этапы 0–5 = MVP** (можно играть с друзьями).
2. **Этап 6** — визуал (можно частично делать параллельно).
3. **Этап 7** — расширения для роста.
